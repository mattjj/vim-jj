INTRODUCTION                                    *jj* *jj.vim*

This plugin provides a small, fugitive.vim-flavored interface to the
Jujutsu version control system (jj).  It works in any jj workspace,
including secondary workspaces created with `jj workspace add`, where no
.git directory exists anywhere above the file: workspace discovery only
ever looks for a .jj directory, and every jj invocation passes an explicit
--repository flag, so Vim's current directory never matters.

Requires Vim 8.2+ (or Neovim) and a reasonably recent jj (tested with
0.43; `jj file annotate -T` support is required for :J blame).

==============================================================================
COMMANDS                                        *jj-commands*

                                                *:J* *:JJ*
:J                      Show `jj status` output in a scratch window.
                        `:JJ` is an alias for every form of the command,
                        for the case where :J is taken by another plugin.

:J {args}               Any subcommand not listed below is passed straight
                        through to jj, and its output is shown in a scratch
                        window, e.g. `:J log`, `:J op log`, `:J new`.
                        Output is rendered with jj's own terminal colors
                        (see |jj-colors|).  Commands that would open an
                        editor should be given `-m` style flags instead
                        (e.g. `:J describe -m message`).  Escape spaces in
                        arguments with a backslash: `:J describe -m two\ words`

                        Maps inside output windows:
                          q       close the window
                          R       re-run the command and refresh
                          <CR>    open the commit whose change id appears
                                  on the current line (great in :J log)
                          o       ... in a horizontal split
                          O       ... in a new tab
                          gf      open the file the command was run from,
                                  as of the commit on the current line

                                                *:J-blame*
:J blame                Analogous to fugitive's :Git blame.  Run
                        `jj file annotate` on the current file and show the
                        annotations in a scroll-bound vertical split to the
                        left.  Annotation runs asynchronously (jj can take
                        a while on files with deep history): the pane
                        opens immediately and fills in when jj finishes.
                        Results are cached per commit - in memory and on
                        disk (see |g:jj_cache_dir|) - so re-blaming an
                        unchanged file is instant, including across Vim
                        restarts and between concurrent Vim instances.
                        Editing the file (or anything else that changes @)
                        invalidates the cache automatically: entries are
                        keyed by immutable commit id, so they can never be
                        stale.  In a |jj-object-buffer| for a file at an old
                        revision, annotates as of that revision.  Extra
                        arguments are passed to `jj file annotate`.
                        Running it again (from either window) closes the
                        blame window.

                        Maps inside the blame window:
                          q       close the blame window
                          <CR>    close blame; show the commit that
                                  introduced the line in the main window
                          o       show that commit in a horizontal split
                          O       show that commit in a new tab
                          gf      close blame; show the blamed file as of
                                  that commit in the main window

                                                *:J-diff*
:J diff [args]          Analogous to fugitive's :Git diff.  Show `jj diff`
                        output (in --git format unless you pass a format
                        flag yourself) in a scratch window with diff
                        highlighting.  E.g. `:J diff`, `:J diff -r @-`,
                        `:J diff --stat`.

                                                *:J-diffsplit*
:J diffsplit [revset]   Analogous to fugitive's :Gdiffsplit.  Vimdiff the
                        current file against the same file at [revset].
                        With no argument, diffs against the parent of the
                        buffer's revision: for a working-copy file that is
                        @- (i.e. what `jj diff` would show for this file);
                        for a file at revision X in a |jj-object-buffer|,
                        it is X-.  Use |:diffget| / |do| to pull hunks from
                        the historical version, and :diffoff if you close
                        things in an unexpected order.  Vim's native |]c|
                        and |[c| jump between changes, as in any diff view.

                                                *:J-diffsplit!*
:J diffsplit!           Analogous to fugitive's :Gdiffsplit! on a
                        conflicted file: a three-pane merge view.  See
                        |jj-merge|.

                                                *:J-edit*
:J edit {object}        Analogous to fugitive's :Gedit.  Open a buffer for
                        a jj object in the current window.  See
                        |jj-object|.  With no argument, from a
                        |jj-object-buffer| for a file, return to the
                        working-copy version of that file.

:J split {object}       Like :J edit, but open in a horizontal split.
:J vsplit {object}      ... vertical split.
:J tabedit {object}     ... new tab.
:J pedit {object}       ... preview window.

                                                *:J-show*
:J show [args]          `jj show --git` output in a scratch window with
                        commit/diff highlighting.

                                                *jj-]c* *jj-[c*
In buffers that display patches (:J diff, :J show, and commit object
buffers), ]c and [c jump to the next/previous @@ hunk header, as in
fugitive's buffers.  In diff-mode windows (:J diffsplit and the
|jj-merge| view) they retain Vim's native jump-between-changes behavior.

==============================================================================
RESOLVING CONFLICTS                             *jj-merge* *jj-conflicts*

jj materializes conflicts as marker blocks in the working copy, and a file
counts as resolved as soon as it contains no markers the next time jj
snapshots the working copy (i.e. on your next jj command) - there is no
"mark as resolved" step like `git add`.

On a conflicted file, `:J diffsplit!` opens a three-pane merge view, like
fugitive's :Gdiffsplit!:

    [side #1]      [working file]      [side #2]

The middle pane is the real file, markers and all; the outer panes are
read-only reconstructions of the two sides of the conflict (their commits
are echoed when the view opens).  All three are in diff mode, so Vim's
native |]c| / |[c| jump between conflict regions, exactly as in fugitive.

Maps, mirroring fugitive:

    d2o     (middle pane) pull the conflict region from side #1
    d3o     (middle pane) pull the conflict region from side #2
    dp      (side panes)  push the region into the middle pane
    q       (side panes)  close the pane

You can also edit the middle pane by hand (e.g. delete marker lines,
combine both sides).  When it's the way you want it, |:w| it; the next jj
command (`:J`, `:J resolve --list`, ...) snapshots the working copy and
the conflict is gone.  Closing the last side pane takes the middle window
back out of diff mode.

Conflicts with more than two sides (rare, from octopus-y histories) are
not supported by the three-pane view; use `jj resolve` or edit the
markers directly.

SPECIFYING OBJECTS                              *jj-object*

:J edit and friends accept the following, echoing fugitive's rev:path
syntax but with jj revsets:

    {revset}            The current file as of {revset}.  Any revset
                        that resolves to a single commit works: `@-`, a
                        change id, a bookmark, etc.  From a buffer with
                        no file (an output window, a commit view), the
                        commit itself instead: description plus --git
                        diff, like `jj show`.
    {revset}:{path}     The file {path} (relative to the workspace root)
                        as of {revset}, e.g. `:J edit @--:src/main.rs`
    {revset}:           The current file as of {revset}
    {revset}:%          Ditto
    :{path}             The file {path} in the working-copy commit @

If the whole argument parses as a revset it is treated as one, so revsets
containing colons still work without quoting.

Object arguments complete on the command line: bookmarks plus common
revsets (@, @-, root(), trunk(), ...).

                                                *jj-object-buffer*
Object buffers have names like
    jj://{root}//{commitid}/{path}
    jj://{root}//{commitid}
and are read-only snapshots produced by `jj file show` / `jj show`.
Revsets are resolved to full commit ids before the buffer is created.
:J blame, :J diffsplit, and :J edit all work from inside a file object
buffer, interpreted relative to that buffer's revision.

COLORS                                          *jj-colors*

jj's terminal output is colorful, and vim-jj keeps it that way: output
windows (:J, :J log, :J op log, ...) run jj with --color=always, strip the
ANSI escape codes, and re-apply them as text properties (Vim 8.2+) or
extmarks (Neovim).  The log graph, bookmark names, timestamps, and jj's
bold shortest-unique-prefix change ids all render exactly as in the
terminal.  Your jj color configuration (in jj's own config.toml) is
respected, since the colors come from jj itself.  Highlight groups are
generated on demand and named jjAnsi_*; set g:jj_color = 0 to disable and
get plain text.

:J blame gives each change id a stable rotating color (like fugitive's
hash colors); these groups are named jjBlameId_{changeid}.

==============================================================================
STATUSLINE                                      *jj-statusline* *jj#Statusline()*

Add the working-copy change id to your statusline: >
        set statusline+=%{jj#Statusline()}
<
It renders like `[jj:tuxqvyvp+]`: the shortest change id of @, with "!"
appended if @ has conflicts and "+" if it is non-empty.  The value is
cached for a few seconds per workspace and computed with
--ignore-working-copy, so it never snapshots, blocks, or takes the
workspace lock; any :J command invalidates the cache immediately.

CONFIGURATION                                   *jj-configuration*

g:jj_executable                                 *g:jj_executable*
        Name or path of the jj binary.  Default: "jj"

g:jj_color                                      *g:jj_color*
        Set to 0 to disable ANSI color rendering in output windows.
        Default: 1

g:jj_blame_async                                *g:jj_blame_async*
        Set to 0 to make :J blame block until jj finishes instead of
        filling the window in asynchronously.  Default: 1

g:jj_cache_dir                                  *g:jj_cache_dir*
        Where blame annotations are cached on disk.  By default each
        workspace caches under its own .jj/vim-jj/cache directory - .jj
        is never tracked, so nothing needs adding to .gitignore (this is
        the same trick fugitive uses with .git).  Set to an absolute path
        to use one shared directory instead, or to '' to disable the disk
        cache.  At most 50 entries are kept, oldest evicted.

                                                *jj#BlameWait()*
jj#BlameWait([{timeout}])
        Wait (up to {timeout} seconds, default 15) for a pending
        asynchronous blame to fill in.  Returns 1 when nothing is pending.
        Useful in scripts and tests.

g:jj_blame_template                             *g:jj_blame_template*
        jj template used for each annotation line in :J blame.  It should
        emit exactly one line per file line and must end with "\n"; it
        should NOT include the line's content.  See the default in
        autoload/jj.vim.
